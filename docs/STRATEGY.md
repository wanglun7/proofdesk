# Proofdesk 产品战略

> **文档性质：** 公司级产品战略，长期维护。每次重大决策、功能方向调整后同步更新。
> **最后更新：** 2026-03-29

---

## 一、产品定位

**我们在做什么：**
Proofdesk 是面向中国市场的企业级合规问卷自动化平台。核心价值是把"收到客户问卷 → 人工逐题填写 → 交付回客户"这条耗时耗力的流程，转化为"上传问卷 → AI 从知识库起草答案 → 人工审核 → 一键导出"的标准化 workflow。

**对标参照：**
海外有 Responsive、Loopio、Conveyor、SecurityPal，覆盖 RFP / Security Questionnaire / DDQ 场景，均已是亿级美金规模公司。中国市场目前无直接对标竞品。

**我们不做什么（当前阶段）：**
- 不做通用 chatbot / 知识库问答
- 不做重实施的咨询交付模式
- 不做需要大量人工运营的服务化方向

---

## 二、核心战略判断

### 2.1 产品死亡路径（逆向推导优先级的依据）

这类产品的共同死法：
> 答案质量下滑 → 用户开始不信任系统 → 全部手动复核 → 退化为"AI 辅助打字员" → 流失

所以所有优先级决策都围绕一个问题：**如何防止用户对系统失去信任？**

优先级不是"对标竞品补功能"，而是**沿着死亡路径，逆向找断点，按断点危险程度排序**。

### 2.2 真正的护城河是数据积累，不是功能数量

Library 里有 10,000 条经过人工审核的标准答案，竞品 UI 再花哨也追不上。
→ 每个功能设计都要问：**这个功能会不会让 Library 变得更丰富、更可信？**

### 2.3 中国市场的独特机会

海外产品深度覆盖 SOC2 / ISO27001，但对中国市场的标准完全空白：
- **等保 2.0**（网络安全等级保护）
- **PIPL**（个人信息保护法）
- **国标 GB/T 系列**
- **央企/国企采购合规问卷**

抢先预置这些模板，是在海外竞品无法直接复制的差异化阵地。

### 2.4 产品演进三代框架（参照海外市场）

| 代际 | 核心卖点 | 代表产品 |
|------|----------|----------|
| 第 1 代 | Proposal / RFP 自动化，Office 集成 | Qvidian |
| 第 2 代 | Response Management，内容库 + 协作工作台 | Loopio, Responsive |
| 第 3 代 | AI-native Trust Platform，Portal 自动化，客户自助 | Conveyor, SecurityPal |

**我们现在处于：** 第 2 代早期入口，核心问答主链已打通，下一步补内容治理和协作，构建第 2 代完整能力后再向第 3 代延伸。

## 三、当前能力快照

> 每次有重大功能上线，在此更新"已完成"状态。

### 已完成 ✅

- 登录认证（JWT），多 Workspace 多租户隔离
- 文档上传（PDF / DOCX / TXT），SSE 流式进度
- 知识库：BM25 + vector hybrid 检索 + Alibaba gte-rerank 重排
- 问卷解析（Excel / TXT），LLM 辅助识别题目列和答案列
- 逐题 AI 起草答案（SSE 流式），含引用 citations
- 人工审核：编辑、Approve、Flag、单题 Regen
- 置信度算法：50% top_rerank×2 + 20% top3_avg×2 + 30% hedge 检测
- Export Excel（摘要表）+ Fill Template（回填原件）
- Answer Library：跨项目复用已审核答案
- 多用户基础架构：Workspace / Account / Membership 模型（Owner / Member 角色）

### 尚不具备 ❌

- Answer Library 治理（owner / 有效期 / 版本 / 状态）
- 题目级 assignment 和协作（指派给成员 / 评论 / @ 通知）
- 问卷完成后内容回流（建议入 Library）
- 标准问卷模板库（等保 / SOC2 / ISO27001 / PIPL）
- Audit Trail（操作历史追踪）
- Analytics（命中率 / 置信度趋势 / turnaround time）
- Trust Center / 外部门户
- Portal 自动化 / 浏览器扩展

## 四、产品路线图

### Phase 1：让核心闭环可信赖（2026 Q2）

**核心命题：** 用户凭什么相信这个答案？

**优先级逻辑：** Library 不可信 → 产品不可信 → 流失。这是最紧迫的死亡断点。

#### P1.1 Answer Library 治理
- 每条 Entry 增加：owner（绑定到 Workspace 成员）、valid_until（有效期）、tags（分类标签）、status（active / needs_review / deprecated）
- usage_count 追踪（被引用几次）
- Library 浏览页支持按 tag / status 筛选、搜索
- **为什么现在做：** 这是整个产品可信度的地基。Library 一旦有过期/错误条目没被发现，AI 答案质量会静默下降。

#### P1.2 答案置信度可解释性增强
- hover 展示置信度低的原因（hedge phrase 触发 / rerank score 偏低 / 来源覆盖不足）
- citation 展示原文高亮位置（不只是"来源文件名 p.X"，还要展示具体命中的文字块）
- **为什么现在做：** 用户看到 "Review — 62%" 但不知道为什么，会开始不信任系统。能解释 = 能信任。

#### P1.3 Audit Trail（操作历史）
- 记录：谁、什么时间、对哪道题、做了什么操作（编辑 / approve / flag / regen / export）
- 问卷级和题目级都可查看操作记录
- **为什么现在做：** 合规场景的硬需求。企业采购方要能证明"答案经过人工审核"，光有 AI 起草是不够的。

#### P1.4 标准问卷模板库
- 预置常见问卷框架：等保 2.0 / SOC2 / ISO27001 / PIPL / GB/T 35274
- 新建问卷时可选模板，直接加载标准题目集
- 模板题目可绑定 Library 推荐答案
- **为什么现在做：** 这是中国市场的独特差异化点，海外竞品完全没有这个方向。早占早建壁垒。

---

### Phase 2：让团队能一起用（2026 Q3）

**核心命题：** 不只是我一个人用，是我们整个团队在跑 compliance workflow。

**优先级逻辑：** 无协作 = 无法进企业采购 = 天花板在个人/小团队。这是营收规模的瓶颈。

#### P2.1 多用户协作工作台
- 问卷级 assignment：整份问卷指定负责人
- 题目级 assignment：每道题指定回答者（可以是非项目负责人的 SME）
- 被分配成员的"我的待办"视图：看到所有分配给自己的题目
- **为什么现在做：** 真实企业里填 security questionnaire 是多角色协作的：产品发起、安全填技术题、法务填隐私题。这个流程现在完全在系统外（微信群 / 钉钉），必须拉回来。

#### P2.2 题目级评论与讨论
- 每道题可发评论、@ 成员
- 评论支持 DingTalk / 企业微信 通知推送（后期）
- **为什么现在做：** 协作过程中的沟通是关键，如果讨论在系统外发生，产品就只是"查看工具"而不是"协作平台"。

#### P2.3 问卷项目进度视图
- 截止日期设置
- 完成度：N/M 道已 approved
- 当前卡点：有几道 flagged / 未处理
- 按成员查看各自完成情况
- **为什么现在做：** 企业管理者需要知道"这份问卷能不能按时交"，没有进度视图就无法管理 deadline。

#### P2.4 问卷完成后内容回流（Content Backflow）
- 完成导出后，系统分析本次新产生的高质量 Approved 答案
- 提示：「本次有 8 道题的答案暂未入库，建议添加到 Library」
- 一键批量入库 + 允许编辑再入库
- **为什么现在做：** 这是产品飞轮的核心。用得越多 → Library 越强 → 答题越准 → 越省人工 → 用得越多。没有回流，产品价值不会随时间增长。

---

### Phase 3：可量化、可扩展（2026 Q4）

**核心命题：** 我能向老板证明这个系统节省了多少时间，以及它越来越聪明。

**优先级逻辑：** 续费和涨价需要可量化的 ROI 证据。没有数据，采购方无法在内部为续费做审批。

#### P3.1 Analytics Dashboard
- 核心指标：平均 turnaround time（从上传到导出）、Library 命中率（答案来自库 vs 从头生成）、平均置信度趋势、review 耗时
- 内容健康度：Library 中 needs_review / deprecated 条目占比、高频被问但无高置信答案的题型
- 用量统计：问卷数 / 题目数 / 已 approved 答案数
- **为什么现在做：** 这是企业续费决策的依据，也是我们向更高价格层级跨越的论据。

#### P3.2 Intake 入口扩展
- Word (.docx) 格式问卷支持
- 邮件附件入口（解析转发到系统的附件）
- 零散问题模式（不是整份问卷，而是粘贴 5-10 个单独问题快速获取答案）
- **为什么现在做：** 现实中不是所有问卷都是 Excel，这会消除部分采购阻力。

#### P3.3 深度 Export 支持
- Word 格式导出
- 自定义封面页、品牌 logo
- 导出历史记录（哪次导出了什么版本）
- **为什么现在做：** 部分客户要求以 Word 格式提交，或要求有公司抬头，这是进大客户的门槛。

---

### Phase 4：第二曲线（2027+）

**核心命题：** 我的客户不用给我发 Excel 了，他们自己就能得到答案。

> 这个阶段是从"Response Management 工具"向"Trust Platform"演进，节奏取决于 Phase 1-3 的市场验证结果。

#### P4.1 Trust Center（外部客户自助门户）
- 企业可发布自己的 Trust Center 页面（对外的安全信息门户）
- 访客可查看：认证证书、安全白皮书、隐私政策、合规文档
- 访客可提问，AI 从 Approved Library 中回答（不暴露内部知识库原文）
- NDA 签署 + 文档访问控制（需要身份验证才能下载）
- 访问审计日志
- **参照：** Responsive Trust Center、Conveyor Trust Center

#### P4.2 Portal 自动化 / 浏览器扩展
- 支持检测第三方 portal 中的问卷表单（如政府采购平台、大型企业供应商门户）
- AI 自动匹配答案并填写，人工确认后提交
- 中国特殊情况：需要适配国内主流 portal（如中央政府采购平台、阿里云市场等）
- **注意：** 中国 portal 标准化程度低于海外，实现复杂度高，等市场成熟再做。

#### P4.3 Vendor Assessment 反向场景
- 不只是"回答别人的问卷"，还能"发问卷给供应商 + 收集回答 + AI 汇总评估"
- 适合企业自身做供应链风险管理

## 五、定价战略参考

> 详细定价策略在单独文档中维护，此处记录核心逻辑。

**定价锚点：**
| 海外参照 | 年费区间 |
|----------|----------|
| Loopio | $30,000 - $100,000+/yr |
| Responsive | $50,000 - $200,000+/yr |
| Conveyor | $20,000 - $80,000+/yr |

**中国市场初步判断（待验证）：**
- Pilot 阶段：¥19,800 / 2个月（验证产品价值，降低采购决策门槛）
- Starter：¥49,800 / 年（小团队，≤3人，基础功能）
- Growth：¥99,800 / 年（团队协作，≤10人，全功能）
- Enterprise：¥199,800+ / 年（无限用户，定制模板，优先支持）

**定价哲学：** 前期低价换用户和 Library 数据，数据积累后提价。初期不要追利润，追 Library 质量。

## 六、关键决策记录

> 每次做出重大产品/战略决策时，在此追加记录，防止团队失忆。

| 日期 | 决策 | 理由 |
|------|------|------|
| 2026-03 | 采用 Tailwind CSS + lucide-react 企业化 UI | 找真实用户测试前需要有足够说服力的视觉专业度 |
| 2026-03 | 置信度算法改为 ×2 线性缩放（去掉 sigmoid） | Alibaba gte-rerank 返回 [0,1] 概率值，sigmoid 会压缩分布导致全部显示 Review |
| 2026-03 | 多租户架构：引入 Workspace / Account / Membership | 为团队协作和 SaaS 多租户扩展打基础，早期做成本低 |
| 2026-03 | Export 统一为"填回原件"，不再单独提供摘要 Excel 按钮 | 简化用户决策路径，有原件填原件，无原件降级输出摘要，一个按钮搞定 |
